    Form Launch URLs and other forms4health APIs

    This page documents form launch URLs and other APIs of forms4health. Note: in any given deployment not all of these APIs will be availble or directly accessible. Likewise depending on the security model many of these APIs will require authentication and/or will be unavailable.

    What you can find on this page:

    • Form Load URL Structure
    • Form Template APIs
    • Form Instance APIs
    • Reporting APIs
      • Form level data extraction
      • Submissions Count
    • Error handling and standard HTTP response codes

    Load URL Structure

    Non-persisted

    Single Pass Forms - Preview in design time environment

    Route: GET /SinglePassDemo/{formKey}/{formVersion}/Load
    Description: Loads a SinglePass form in a demo (when submitted renders resulting PDF, no data is stored)
    Example: /SinglePassDemo/dental-smile-assessment/1/Load

    Single Pass Forms

    Route: GET {tenantId}/SinglePass/{formKey}/{formVersion}/Load
    Description: Loads a SinglePass form for the associated tenant
    Example: {tenantId}/SinglePass/dental-smile-assessment/1/Load

    Single Pass Forms (customer URL)

    Typically used in conjunction with a proxy server to provide seamless integration with CMS such as wordpress.
    Route: GET {organisation forms domain/subdomain}/forms/{formKey}/{formVersion}
    Description: Loads a SinglePass form in the context of an organisation’s own domain name
    Example: https://forms.orgdomainname.com/forms/sick-note-request/1 - Sub-domained
    Example: https://gp-practices.co.uk/forms/sick-note-request/1 - Proxied

    Create form instances for a given patient

    Route: POST /api/clinicianforms/create/
    Permission: Create
    Description: Create an instance of a form for a given patient

    Request

    {
    PatientIdentifier: "a4847fd9-7da1-49da-b5f4-be278dba0730",
    InstanceIdentifier: "70b9802a-c7d2-420c-bdc1-0a32b6d06c19",
    FormKey: "end-of-life-care",
    FormVersion: 1
}

    Withdraw a form instance

    Route: DELETE /api/patientforms/{PatientId}/instance/{InstanceId}/withdraw
    Permission: Withdraw
    Description: Marks a form instance and all its instance versions as being withdrawn and therefore removing it from any API responses unless explicitly requested with the query parameter includeWithdrawn=true

    Request

    {
    "Reason": "Form created against wrong patient by mistake"
}

    Reinstate a form instance

    Route: PATCH /api/patientforms/{PatientId}/instance/{InstanceId}/reinstate
    Permission: Reinstate
    Description: Marks a form instance and all its instance versions as being reinstated and therefore allowing it to be included in any API responses.

    Request

    {
    "Reason": "Form withdrawn in error"
}

    Reporting APIs

    The following APIs are used to extract data from forms4health for the purposes of reporting on persisted clinician forms. A report definition can be constructed to query forms4health for data at a form or building block level. The data is returned in JSON with a table (columns and rows) like structure.

    Form level data extraction

    Route: POST /reporting?format={format}
    Permission: RunExtractsAndReports
    Description: Returns data extracted using a report definition query
    Example: /reporting?format=google

    The format query parameter is optional and the currently only supported option is google which is for use in Google Charts, if not included data is returned in plain json with a columns and rows breakdown.

    In order to extract data from forms4health a reporting definition needs to be sent with the request, this definition can be thought of as a query, it also allows us to define which, form & building blocks are included as well as limiting results to a date range and row count.

    Reporting definition

    {
    "PatientIds": [
        "488edeAB-5e8F-4e50-9137-24e377cf9436"
    ],
    "ReportDefinition": {
        "type": "continuous",
        "series": [{
            "label": "Number of cigarettes per day",
            "dataPointId": "number-per-day"
        }, {
            "label": "Smoker",
            "dataPointId": "is-a-smoker"
        }],
        "extractDefinition": {
            "type": "form",
            "key": "smoking-assessment",
            "version": "1",
            "range": {
                "from": "2020-01-01",
                "to": "2020-06-30",
                "maxResults": 100,
                "maxResultsPerSubject": 10
            },
            "dataPoints": [{
                "id":  "number-per-day",
                "controlId": "number-per-day",
                "buildingBlock": {
                    "key": "smoking",
                    "version": "1"                    
                }
            }, {
                "id":  "is-a-smoker",
                "controlId": "is-a-smoker",
                "buildingBlock": {
                    "key": "smoking",
                    "version": "1"                    
                }
            }]
        }
    }
}

    To restrict the results to certain patients we can provide a list of required patient identifiers, if the value is ommitted data for all patients will be returned where that data exists.

    "PatientIds": [
    "488edeAB-5e8F-4e50-9137-24e377cf9436"
]

    There are two types of reports, discrete and continuous. Discrete data is that which is counted and cannot be divided, it can be plotted on a bar graph. Examples of discrete data are:

    Number of students
    Amount of money

    Continuous data is that which can be measured and can be dividied infinitely, it can be graphed on a histogram or line graph e.g.

    Weight of a patient
    Height of a patient
    Time it takes to run a mile

    We can set which data type we are using by setting ReportDefinition.Type

    "ReportDefinition": {
    "type": "continuous",

    We can define the data series used in our report, the label field will control the name of the column returned and the dataPointId must match one of the data points defined in the extraction definition below.

    For continuous data we must provide an array of series objects

    "series": [{
    "label": "Number of cigarettes per day",
    "dataPointId": "number-per-day"
}, {
    "label": "Smoker",
    "dataPointId": "is-a-smoker"
}],

    For discrete data we must provide a single series object

    "series": {
    "label": "Number of cigarettes per day",
    "dataPointId": "number-per-day"
}

    The extract definition helps us build a query for extracting the data. We can extract data at form level with multiple building blocks or at an individual building block level. This example demonstrates form level extraction.

    type = form or building-block
    key = form name
    version = form version

    "extractDefinition": {
    "type": "form",
    "key": "smoking-assessment",
    "version": "1"

    We can restrict the data returned to either a date range based on form submission date or a hard limit on the number of results

    maxResults = limit the overall number of results
    maxResultsPerSubject = limit number of results when data is partitioned by patient, building block name and version

    "range": {
    "from": "2020-01-01",
    "to": "2020-06-30",
    "maxResults": 100,
    "maxResultsPerSubject": 10
},

    We can define what data is actually returned as part of our report by providing an array of data points.

    id = identifier for each data point
    controlId = identifer for the control we wish to extract as defined in the building block
    buildingBlock.Key = Name of the bulding block the control resides in
    buildingBlock.Version = Which version of the building block to use

    "dataPoints": [{
    "id":  "number-per-day",
    "controlId": "number-per-day",
    "buildingBlock": {
        "key": "smoking",
        "version": "1"                    
    }
}, {
    "id":  "is-a-smoker",
    "controlId": "is-a-smoker",
    "buildingBlock": {
        "key": "smoking",
        "version": "1"                    
    }
}] # Error handling and standard HTTP response codes

    HTTP Response Codes are used per standard REST API design. However the most commonly used codes by forms4health are listed below. a Where APIs used a different set or response codes / behaviours these are individually documented with the API

    Code Description
    200 Success - The operation to load/save a form or call an API succeded. The response payload will be dependent on the URL/API called
    400 Bad Request - Occurs if the API request payload (or URL parameters) fail validation or are otherwise malformed
    403 Forbidden - The User does not have the correct permissions (or is not authenticated), and/or the tenantId or formKey/version configuration does not support the requested operation
    404 Not Found - The requested URL / API does not exist and/or the tenantId and/or patientId does not exist and/or the formKey/formVersion does not exist or is not in a published state
    500 Internal Server Error - An error has occured on the server
    503 Service Unavailable - The server might be re-starting or (more likely) a proxy server cannot contact the service
